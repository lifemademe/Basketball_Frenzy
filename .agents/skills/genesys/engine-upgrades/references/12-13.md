# 12 → 13

## 1. `Pawn` split into a layered hierarchy

`Pawn` no longer bundles movement, combat, interaction, and raw input delegates. Pick the shallowest replacement class that matches what your pawn actually uses:

- `Pawn` — possession only. No movement, no input, no combat/interaction. `movementComponent` is no longer an accepted option.
- `MovementPawn` — `Pawn` + `movementComponent` + action methods (`moveForward`, `moveRight`, `lookUp`, `lookRight`, `zoom`, `jump`, `stopJump`).
- `GameplayPawn` — `MovementPawn` + combat (`fire`, `endFire`, `altFire`, `endAltFire`, `reload`) + interaction (`interact`, `endInteract`, `getInteractionComponent`). No camera/mesh scaffold.
- `CharacterPawn` — `MovementPawn` + the camera/animation/mesh scaffold (first- or third-person). No combat/interaction.
- `DefaultCharacterPawn` — `CharacterPawn` + the same combat/interaction as `GameplayPawn`, plus the pre-13 raw `onKeyDown`/`onKeyUp`/`onMouseDown`/`onMouseUp`/`onGamepadButtonDown`/`onGamepadButtonUp`/`onGamepadAxisChange` delegates.

```ts
// Before (12): CharacterPawn had fire/interact/onKeyDown built in
class MyPlayer extends ENGINE.CharacterPawn {
  protected override doBeginPlay(): void {
    super.doBeginPlay();
    this.onKeyDown.add((e) => { /* ... */ return false; });
  }
}

// After (13): use DefaultCharacterPawn to keep the same members
class MyPlayer extends ENGINE.DefaultCharacterPawn {
  protected override doBeginPlay(): void {
    super.doBeginPlay();
    this.onKeyDown.add((e) => { /* ... */ return false; });
  }
}
```

If a pawn only used `Pawn.create({ movementComponent: null })` for possession with no movement at all, drop the option and extend `Pawn` directly.

## 2. Movement/action methods renamed

| Before (12) | After (13) |
|---|---|
| `addForwardInput` | `moveForward` |
| `addRightInput` | `moveRight` |
| `addLookUpInput` | `lookUp` |
| `addLookRightInput` | `lookRight` |
| `addZoomInput` | `zoom` |
| `handleJump` | `jump` |
| `handleStopJump` | `stopJump` |
| `beginFire` | `fire` |
| `beginAltFire` | `altFire` |
| `beginInteract` | `interact` |

`endFire`, `endAltFire`, `endInteract`, `reload`, and `getInteractionComponent` are unchanged.

## 3. `PlayerController` split into identity/possession base + `DefaultPlayerController` input scheme

`PlayerController` no longer translates raw WASD/mouse/gamepad/joystick input into pawn movement. It's now a minimal base that only owns player identity (`PlayerInfo`), map-transition RPCs, and input-handler registration. That translation moved to a new `DefaultPlayerController` subclass.

- If your game extended `PlayerController` directly in 12 and got WASD/mouse/gamepad movement for free, extend `DefaultPlayerController` in 13 instead.
- If your game already fully replaced the built-in movement translation (only used `PlayerController` for identity/possession and wrote its own input handling), no change needed.

```ts
// Before (12): PlayerController drove WASD/mouse/gamepad movement out of the box
class MyController extends ENGINE.PlayerController {}

// After (13): extend DefaultPlayerController to keep the same built-in movement scheme
class MyController extends ENGINE.DefaultPlayerController {}
```

`GameBuilder`/`GameMode` defaults, all demos, and other SDK skill docs were updated to reference `DefaultPlayerController` accordingly — check any code that reads `ENGINE.PlayerController` expecting movement behavior.

## 4. New abstract `Controller` base class

`PlayerController` now extends a new abstract `Controller` base class, which owns possession bookkeeping (`possess`/`unpossess`), `controlRotation`, and `isLocalController()`. As a result:

- `Pawn.getController()` now returns `Controller | null` instead of `PlayerController | null`.
- `Pawn.getPlayerController()` is deprecated but still works as a shim — narrow with `instanceof ENGINE.PlayerController` (or check `isPlayerControllerType()`) if you need `PlayerController`-specific members and were relying on the old return type.

```ts
// Before (12)
const playerController = pawn.getPlayerController();

// After (13)
const controller = pawn.getController();
const playerController = controller instanceof ENGINE.PlayerController ? controller : null;
```

## 5. Raw input handling moved to `PlayerController`

`Pawn`/`MovementPawn`/`CharacterPawn`/`GameplayPawn` no longer have `onKeyDown`/`handleKeyDown`/`handleMouseDown`/`handleMouseWheel`/`handleGamepadButtonDown`/etc. Only `DefaultCharacterPawn` still exposes the raw delegates, for migration. New pawn-level input handling should move into a `PlayerController` subclass:

```ts
// Before (12): input handled on the pawn
class MyPlayer extends ENGINE.CharacterPawn {
  public override handleKeyDown(e: KeyboardEvent): boolean {
    if (e.key === 'e') { this.tryInteract(); return true; }
    return super.handleKeyDown(e);
  }
}

// After (13): input handled on the controller
class MyPlayerController extends ENGINE.DefaultPlayerController {
  public override handleKeyDown(e: KeyboardEvent): boolean {
    if (e.key === 'e' && this.pawn instanceof MyPlayer) {
      this.pawn.tryInteract();
      return true;
    }
    return super.handleKeyDown(e);
  }
}
```

## 6. Custom `PlayerController`s must narrow `this.pawn` before calling action methods

A possessed `this.pawn` is no longer guaranteed to have movement or combat/interaction methods. Guard with `instanceof MovementPawn` for movement, and the structural `isCombatPawn`/`isInteractPawn` guards for combat/interaction (`GameplayPawn` and `DefaultCharacterPawn` intentionally don't share a common ancestor, so `instanceof GameplayPawn` would miss `DefaultCharacterPawn`).

```ts
// Before (12)
this.pawn?.addForwardInput(1, false);
this.pawn?.beginFire();

// After (13)
if (this.pawn instanceof ENGINE.MovementPawn) {
  this.pawn.moveForward(1, false);
}
if (ENGINE.isCombatPawn(this.pawn)) {
  this.pawn.fire();
}
```

## 7. `CharacterPawn.zoom()` override replaced with `zoomStep()`

If a subclass overrode `CharacterPawn.zoom()` or `handleMouseWheel()` to customize spring-arm zoom, override `zoomStep(direction)` instead — it's called once per discrete wheel notch, not every frame. `zoom()` on `CharacterPawn` is now the inherited continuous per-frame axis (forwarded to `movementComponent`, like any other `MovementPawn`).

```ts
// Before (12)
public override zoom(value: number, isAbsolute: boolean = false): void {
  // custom spring-arm zoom
}

// After (13)
public override zoomStep(direction: number): void {
  // custom spring-arm zoom
}
```

Reference implementations: `Pawn.ts`, `MovementPawn.ts`, `GameplayPawn.ts`, `CharacterPawn.ts`, `DefaultCharacterPawn.ts`, `PawnActions.ts`, `Controller.ts`, `PlayerController.ts`, and `DefaultPlayerController.ts` in `.engine/src/actors/`.
