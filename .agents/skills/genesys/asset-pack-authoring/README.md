# Asset Pack Authoring — Background

Reference material for the `asset-pack-authoring` skill. The skill files
(`SKILL.md` and `references/*.md`) contain the operational procedure; this
file holds the rationale and conceptual context the agent does not need
loaded for every invocation.

## Why asset packs exist

An asset pack groups related code and content under `<project>/packs/<name>/`
so a feature can be shared, versioned, or removed as a unit. Packs live next
to the project's own `src/` and `assets/` trees but stay self-contained: a
pack should be removable by deleting its folder.

## Two path namespaces

The skill repeatedly distinguishes two prefixes that look similar but resolve
through completely different mechanisms:

- `@packs/<name>/...` is a TypeScript import alias only. It is wired by
  `tsconfig.compilerOptions.paths` and a per-pack Vite `resolve.alias` entry,
  and resolves to `<project>/packs/<name>/src/...`. It exists only at compile
  time and only inside `import` statements.
- `@project/packs/<name>/assets/...` is a runtime asset URL. The engine's
  `resolveAssetPathsInText` regex recognises only `@engine/` and `@project/`
  at runtime, so pack assets are addressed as siblings of the regular
  `@project/assets/...` tree.

Mixing the two — for example writing `@packs/my-pack/assets/foo.png` in a
string literal — produces a path the engine cannot resolve.

## Class identity is unchanged by a move

A class decorated with `@ENGINE.GameClass()` registers under its class name
into a global registry at startup, regardless of which file or pack it lives
in. Prefab `$bc` values such as `"GAME.MyActor"` therefore stay the same when
`MyActor` moves into a pack. Class-name uniqueness across the whole project
plus all installed packs is the pack author's responsibility; prefix names
if collisions are likely.

## What lives inside `assets/` vs `src/`

- `assets/` — anything fetched at runtime: textures, models, audio, video,
  HDRIs, prefabs (`*.prefab.json`), materials (`*.material.json`), JSON data.
- `src/` — TypeScript modules: `@ENGINE.GameClass`-decorated classes, helper
  functions, registered specializations, side-effect imports.

TypeScript should not live under `assets/`, and runtime assets should not
live under `src/`.

## Common migration failures and remedies

- `pnpm compile` reports "Cannot find module './foo'" in `src/game.ts` —
  step 6 of the checklist missed a caller. Rewrite the import to
  `@packs/<name>/foo`.
- `pnpm compile` reports stale imports in `src/game-data.ts` or
  `src/auto-imports.ts` — both files are regenerated by the build pipeline.
  Run `pnpm build` (or `pnpm run generate-property-metadata` for
  `game-data.ts`) to refresh them; never edit them by hand.
- `validate-prefabs` rejects a prefab — usually a typo introduced during
  rewrite. Check the offending string value.
- `check-pack-isolation` reports an external `@project/assets/...` reference
  inside the pack — either move that asset into the pack (extend the inputs
  and rerun the migration) or accept it as an external dependency and
  document it in `config.json`.
- `check-pack-isolation` reports a stale project reference — a file under
  `src/` or `assets/` still points at a pre-move location. Update or delete
  it. (Auto-generated `auto-imports.ts` and `game-data.ts` are skipped by
  the scan; regenerate them via the build pipeline instead.)
